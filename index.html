Abstract

Communication barriers remain a significant challenge in creating accessible and inclusive environments for individuals who rely on sign language. As a proof of concept toward reducing these barriers, we study American Sign Language (ASL) fingerspelling recognition as static letter classification using a CLIP-based image encoder. We fine-tune a CLIP ViT-B/32–based model and evaluate it in strict held-out validation and test modes. On a held-out validation run (844 samples), the fine-tuned checkpoint achieves 99.29% top-1 accuracy and 99.88% top-5 accuracy, with very high average confidence (0.9957). However, on a cross-dataset test run (10,033 samples), performance drops to 33.69% top-1 accuracy and 62.65% top-5 accuracy, with substantially lower average confidence (0.6333), highlighting the impact of distribution shift on generalization. We include a zero-shot CLIP baseline (openai/clip-vit-base-patch32), which performs poorly on the same test setting (5.39% top-1, 21.89% top-5, avg confidence 0.0666), reinforcing the need for task-specific adaptation.

Method

Task

We treat ASL fingerspelling recognition as a multi-class image classification problem. Given a single RGB image of a handshape, the system predicts the corresponding letter class from a fixed label set used in evaluation.

Dataset and preprocessing

We evaluate on two labeled image datasets: one used for training/validation and a separate dataset used as a cross-dataset test benchmark. Images are processed into a consistent square input format suitable for CLIP ViT-B/32 inference.

Data splits

Training is performed on the training data. Validation is run on a held-out validation subset used for checkpoint selection. Final reporting is run in a strict test mode on the test benchmark. We avoid using the training split for reporting metrics, and we treat cross-dataset evaluation as a domain-shift stress test.

Model: prompt-based CLIP classification

We use a CLIP ViT-B/32–based classifier.

a photo of a hand showing the sign language letter X

For each input image, the model outputs a score distribution over the label set, and the predicted class is the label with the highest score. We report both top-1 and top-5 accuracy to capture whether the correct label appears among the most likely predictions.

Trainable parameters: the fine-tuned checkpoint updates the CLIP vision encoder while using fixed text prompts for classification.

Training objective

We fine-tune using a supervised classification objective over the model’s logits against the ground-truth class label.

Optimization and schedule

We train with AdamW (learning rate 1e-5, weight decay 0.01), batch size 32, for 15 epochs.

Evaluation

  • We report top-1 accuracy,
  • top-5 accuracy,
  • average confidence,
  • correct/total predictions,
  • and per-sample predictions (true label, predicted label, confidence).
  • Where available in the dashboard artifacts,
  • we also visualize confusion matrices and calibration summaries.

Limitations

Held-out validation drawn from the training distribution can overstate real-world performance. Cross-dataset testing is substantially harder and may reflect differences in signers, lighting, backgrounds, framing, camera quality, or label conventions.

Results

We report results for (i) a held-out validation evaluation of the fine-tuned checkpoint and (ii) a cross-dataset test evaluation. We also include a zero-shot CLIP baseline to contextualize the benefit of fine-tuning.

Baseline comparison

Dataset
Baseline comparison (cross-dataset test): We compare a zero-shot CLIP baseline against our fine-tuned checkpoint on the same test benchmark. The fine-tuned model substantially outperforms the zero-shot baseline in both top-1 and top-5 accuracy, indicating that task-specific adaptation is necessary for this domain. Fine-tuned (epoch_9): top-1 0.3369, top-5 0.6265, avg confidence 0.6333 (3,380 / 10,033 correct). Zero-shot (openai/clip-vit-base-patch32): top-1 0.0539, top-5 0.2189, avg confidence 0.0666 (586 / 10,873 correct).

Overall, these results show a clear generalization gap: the fine-tuned CLIP-based model performs extremely well on held-out validation drawn from the same distribution, but degrades sharply when evaluated on a different dataset. The large cross-dataset drop—paired with a substantial reduction in average confidence—suggests that changes in capture conditions and data distribution materially affect recognition performance. The zero-shot baseline further indicates that off-the-shelf CLIP features alone are not sufficient for reliable ASL fingerspelling classification in this setting, and that fine-tuning is critical even though it does not fully solve robustness under domain shift. This motivates future work focused on improving cross-dataset robustness through more diverse training data, better domain coverage, and evaluation protocols that reflect deployment conditions.

Repo & setup notes

Repository: transladoorss-model-dev-env (see Code link). This project is a web-based ASL letter prediction system built around three components: a Django backend API (backend/), a lightweight frontend UI (frontend/), and an ML training/evaluation module (ml/). Project structure: backend/ (Django project + API + backend tests), frontend/ (index.html and JS logic for webcam capture/translate/feedback), ml/ (fine-tuning, checkpoint export, inference helper, evaluator), env/ (env/environment_unified.yaml). Environment & requirements: Python 3.10–3.12; Conda recommended via env/environment_unified.yaml; Node.js for frontend tests; optional GPU acceleration (CUDA/MPS). Run the web app: From backend/, run migrations and start the dev server. In DEBUG, Django serves the frontend at http://localhost:8000/ (frontend/index.html).

  • Local setup: Create and activate the Conda environment, then set environment variables via python-decouple: SECRET_KEY (required), DEBUG=true for local dev, and ALLOWED_HOSTS (e.g., localhost,127.0.0.1). Optional ML configuration: ASL_MODEL_ID (checkpoint path) and ASL_MODEL_DEVICE (cuda/mps/cpu).
  • API endpoints: POST /img_in/translate/ accepts an image plus optional top_k (1–26) and returns either a single prediction (top_k=1) or a top-k list (top_k>1). POST /img_in/feedback/ stores the uploaded frame along with predicted_label and correct_label; in development, images are stored under backend/feedback_images.
  • Model development utilities: Fine-tune via ml.development.clip_fine_tuned; export the best checkpoint via ml.development.export_best_checkpoint; run local inference via ml.inference.service; evaluate via ml.testing.evaluator.
  • Testing: Backend tests run with pytest -c backend/pytest.ini. Frontend tests run with cd frontend/js && npm test. Keep API response contracts consistent across both suites if responses change.

Current gap: Feedback is stored for future retraining, but an automated retraining pipeline is not yet implemented.
