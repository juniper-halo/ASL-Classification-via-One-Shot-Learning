Abstract

Communication barriers remain a significant challenge in creating accessible and inclusive environments for individuals who rely on sign language. As a proof of concept toward reducing these barriers, we study American Sign Language (ASL) fingerspelling recognition as static letter classification. We use a prompt-based CLIP ViT-B/32 setup where each class is represented by a fixed text prompt (e.g., “a photo of a hand showing the sign language letter X”), and fine-tune only the vision encoder while keeping the text encoder and prompts fixed. Models are trained on the dataset’s provided training split, with a held-out validation subset used for model selection. Final evaluation is run in strict validate/test modes (never on train) and produces accuracy/F1, confusion matrices, calibration (ECE), and throughput metrics. While strong in-distribution performance is achievable, we emphasize that generalization can degrade under distribution shift, motivating future work on more diverse data and temporally structured inputs.

Method

Task

We address ASL fingerspelling letter recognition as a multi-class image classification problem. Given a single RGB image of a handshape, the system predicts the corresponding English letter (A–Z), excluding motion-dependent letters when not present in the dataset.

Dataset and preprocessing

We use a Hugging Face ASL letter image dataset with letter labels normalized to a consistent A–Z class set when possible (some datasets omit motion-dependent letters such as J/Z). Images are converted to RGB and deterministically preprocessed into 224x224 square inputs via center-cropping and resizing (LANCZOS). No stochastic data augmentation is applied.

Data splits

Training uses the dataset’s provided train split. For validation, we create a random 90/10 split of the training data using a fixed seed (42). The evaluator supports strict, safe modes only: validate evaluates the held-out validation subset and test evaluates the dataset’s test split (when available). If a required split is missing, evaluation errors rather than falling back to another split, and evaluation on the training split is explicitly disallowed.

Model: prompt-based CLIP classification

We use CLIP ViT-B/32 in a prompt-based classification setup. Each class is represented by a fixed text prompt of the form:

a photo of a hand showing the sign language letter X

For an input image, CLIP produces image-text similarity logits (logits_per_image) over the 26 prompts. The predicted letter is the prompt with the highest logit.

Trainable parameters: only the vision encoder is updated during training. The text encoder and prompts remain fixed, ensuring the model learns visual features aligned to a stable text embedding space.

Training objective

We train using cross-entropy loss computed over the CLIP image-text logits against the ground-truth class label. This directly optimizes the model for correct prompt selection in the 26-way classification setting.

Optimization and schedule

Training uses AdamW (learning rate 1e-5, weight decay 0.01), batch size 32, for 15 epochs, with a linear learning-rate schedule (warmup default 0 steps) and gradient accumulation of 1.

Evaluation

  • Overall and per-class accuracy.
  • Confusion matrix (counts and row-normalized).
  • Top-k accuracy (k=5 when probabilities are available).
  • Per-class and aggregate precision/recall/F1 (macro and micro).
  • Confidence statistics (average confidence).
  • Calibration curve and expected calibration error (ECE).
  • Runtime/throughput measurements.

Limitations

Because validation is constructed via a random split from the training data and preprocessing is deterministic, performance on held-out splits may not fully reflect generalization across unseen signers, environments, or capture conditions. We treat results as a proof of concept for prompt-based CLIP adaptation and emphasize the need for signer-independent and deployment-style evaluation in future work.

Results

We report results on (i) a held-out validation split, (ii) the final test split, and optionally (iii) cross-dataset OOD evaluation when available.

Repo & setup notes

Repository: transladoorss-model-dev-env (Code). In DEBUG, Django serves frontend/index.html plus static /js, /css, /assets, and /media (uploads under backend/feedback_images).

  • Frontend: static Bootstrap + vanilla JS (webcam capture, fetch to API); Jest tests in frontend/tests/scripts.test.js.
  • Backend: Django + DRF; pytest/pytest-django cover API behaviors. MEDIA_ROOT switches to a temp dir during tests.
  • Known gaps: backend/tests/test_img_translate.py has merge-conflict markers; feedback storage exists but automated retraining is not wired up. The evaluator (ml/testing/evaluator.py) is the single source of truth for metrics/artifacts and is invoked in validate mode during training and in test mode for final reporting.
  • Assets: replace static/images/favicon.ico and consider adding a static/images/social_preview.png for richer sharing cards.

You can swap static/pdfs/sample.pdf for an architecture or API reference if you want a downloadable artifact.
